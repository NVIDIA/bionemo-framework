# bmm_ext

`bmm_ext` is an out-of-tree PyTorch CUDA extension that exposes a `torch.bmm`-style API for batched low-precision matmul with explicit block scaling.

Current formats:

- `mxfp8`
- `nvfp4`

The public API is Pythonic. The compute backend is native C++/CUDA.

## What It Does

This package provides:

- `bmm_block_scaled(...)` for `(B, M, K) x (B, K, N) -> (B, M, N)`
- `pack_nvfp4(...)` and `unpack_nvfp4(...)` helpers for explicit NVFP4 packing
- strict validation for shape, device, dtype, and scale contracts

This is not an upstream PyTorch patch and it does not replace `torch.bmm`.

## Current Behavior

- CUDA only
- forward only
- no broadcasting
- explicit scale tensors are required
- output dtype defaults to `torch.float32`
- accumulation is intended to stay in `float32`

Precision behavior:

- `mxfp8` uses FP8 inputs plus explicit block scales
- `nvfp4` uses explicit packed FP4 inputs plus explicit scales
- both are compared against `torch.bmm` FP32 references in the example benchmark

## API

```python
from bmm_ext import PackedNVFP4Tensor, bmm_block_scaled, pack_nvfp4, unpack_nvfp4
```

### `bmm_block_scaled`

```python
bmm_block_scaled(
    a,
    b,
    *,
    a_scale,
    b_scale,
    format,
    out_dtype=torch.float32,
    sf_vec_size=32,
) -> torch.Tensor
```

Rules:

- shape semantics match `torch.bmm` exactly: `(B, M, K) x (B, K, N) -> (B, M, N)`
- batch sizes must match exactly
- inner dimensions must match exactly
- all tensors must be on the same CUDA device
- tensors requiring grad are rejected

### MXFP8 contract

- `format="mxfp8"`
- `a` and `b` must be plain CUDA tensors
- supported input dtypes:
  - `torch.float8_e4m3fn`
  - `torch.float8_e5m2`
- `a_scale` and `b_scale` must use `torch.float8_e8m0fnu`
- `sf_vec_size` defaults to `32`
- `K` must be divisible by `sf_vec_size`

Expected scale shapes:

- `a_scale`: `(B, M, K / sf_vec_size)`
- `b_scale`: `(B, K / sf_vec_size, N)`

The implementation also accepts the padded GEMM-ready scale layouts generated by the backend helper path.

### NVFP4 contract

- `format="nvfp4"`
- `a` and `b` must be `PackedNVFP4Tensor`
- packed storage uses `torch.uint8`
- `sf_vec_size` is currently fixed at `16`
- `K` must be divisible by `32`

Use the helper packer:

```python
a = pack_nvfp4(a_fp32, role="lhs")
b = pack_nvfp4(b_fp32, role="rhs")
```

Important:

- the RHS must be packed with `role="rhs"`
- `a_scale` and `b_scale` are typically `a.scale_inv` and `b.scale_inv`
- `pack_nvfp4(...)` currently relies on Transformer Engine’s NVFP4 quantizer for helper packing

## Example

MXFP8:

```python
import torch
from bmm_ext import bmm_block_scaled

B, M, K, N = 2, 16, 32, 8

a_fp32 = torch.randn(B, M, K, device="cuda", dtype=torch.float32)
b_fp32 = torch.randn(B, K, N, device="cuda", dtype=torch.float32)

a = a_fp32.to(torch.float8_e4m3fn)
b = b_fp32.to(torch.float8_e4m3fn)

a_scale = torch.ones((B, M, K // 32), device="cuda", dtype=torch.float8_e8m0fnu)
b_scale = torch.ones((B, K // 32, N), device="cuda", dtype=torch.float8_e8m0fnu)

out = bmm_block_scaled(
    a,
    b,
    a_scale=a_scale,
    b_scale=b_scale,
    format="mxfp8",
    out_dtype=torch.float32,
    sf_vec_size=32,
)
```

NVFP4:

```python
import torch
from bmm_ext import bmm_block_scaled, pack_nvfp4

B, M, K, N = 1, 16, 32, 16

a_fp32 = torch.randn(B, M, K, device="cuda", dtype=torch.float32).clamp(-6, 6)
b_fp32 = torch.randn(B, K, N, device="cuda", dtype=torch.float32).clamp(-6, 6)

a = pack_nvfp4(a_fp32, role="lhs")
b = pack_nvfp4(b_fp32, role="rhs")

out = bmm_block_scaled(
    a,
    b,
    a_scale=a.scale_inv,
    b_scale=b.scale_inv,
    format="nvfp4",
    out_dtype=torch.float32,
    sf_vec_size=16,
)
```

## Build

Build in place:

```bash
TORCH_CUDA_ARCH_LIST='12.0' python setup.py build_ext --inplace -v
```

Editable install also works:

```bash
pip install -e .
```

The current build expects:

- a working CUDA toolkit
- PyTorch with CUDA support
- Blackwell-class GPU support in the target environment
- Transformer Engine headers and shared library available at the paths used in [setup.py](/workspace/bmm/setup.py)

## Benchmark

The sample benchmark lives at [examples/demo_bmm.py](/workspace/bmm/examples/demo_bmm.py).

Run it with the default large problem size:

```bash
PYTHONPATH=. python examples/demo_bmm.py
```

Default benchmark shape:

- `B=1`
- `M=10240`
- `K=10240`
- `N=10240`

The benchmark:

- times FP32 `torch.bmm`
- times `mxfp8`
- times `nvfp4`
- reports distortion against FP32 references
- prints estimated FP32 live-memory usage

Notes:

- TF32 is disabled for the FP32 reference by default; use `--allow-tf32` to enable it
- NVFP4 distortion is measured against FP32 matmul on the same clamped inputs used for NVFP4 packing
- timing order is hardware- and backend-dependent; measure on the target GPU rather than assuming it

## Layout And Semantics

Logical input/output semantics follow `torch.bmm`:

- `a`: `(B, M, K)`
- `b`: `(B, K, N)`
- `out`: `(B, M, N)`

There is no broadcasting and no implicit repacking in the public API.

## Limitations

- no autograd support
- no CPU support
- no fallback to dense FP16/BF16/FP32 inside `bmm_block_scaled`
- NVFP4 helper packing is explicit rather than hidden inside the matmul call
- the project is currently tuned for NVIDIA-specific low-precision paths rather than portable backends
